Hello world !

In this example, we are going to run a very simple SYCL kernel on a SYCL device on your system using the SYCL programming model. Previous device discovery example didn’t execute anything on a SYCL device as it was just querying about the available SYCL platforms and the SYCL devices. The complete source code for the hello world example is shown below.

 1#include <sycl/sycl.hpp>
 2
 3using namespace sycl;
 4
 5int main(int argc, char **argv) {
 6  const size_t N = (argc > 1 ? std::atoi(argv[1]) : 8);
 7
 8  //! [queue_with_properties_begin]
 9  property_list q_props{property::queue::in_order(),
10                        property::queue::enable_profiling()};
11  sycl::queue   q{q_props};
12  //! [queue_with_properties_end]
13
14  //! [iteration_domain_begin]
15  auto iter_domain = range<1>{N};
16  //! [iteration_domain_end]
17  //! [kernel_begin]
18  auto kernel = [=](sycl::item<1> idx) {
19    ext::oneapi::experimental::printf("Hello from rank = %d\n", idx[0]);
20  };
21  //! [kernel_end]
22  //! [submit_begin]
23  auto event = q.parallel_for(iter_domain, kernel);
24  event.wait();
25  //! [submit_end]
26
27  //! [print_device_begin]
28  std::cout << "The kernel executed on: "
29            << q.get_device().get_info<info::device::name>();
30  //! [print_device_end]
31
32  return 0;
33}

Remember from our earlier discussion that we need to submit SYCL kernels into a SYCL queue in order for them to be executed. SYCL provides several constructors to create customized queues. Below is one of the constructors available for the sycl::queue class (copy pasted from the Intel oneAPI implementation of the SYCL standard). Here prop_list is a list of optional properties used for the queue construction.

queue(const property_list &prop_list = {});

Below is a list of all the properties supported by the Intel oneAPI compiler. The properties under the sycl::ext::oneapi:: namespace are Intel oneAPI extensions.

We pass sycl::property::queue::in_order and sycl::property::queue::enable_profiling to our queue constructor. The enable_profiling property instructs the SYCL runtime to capture profiling information for the kernels that are submitted to the queue. We will see later how to query these profiling information. The in_order property instructs SYCL runtime to schedule the SYCL kernels in-order they are submitted to the queue. Note that these properties are implemented as C++ classes and hence we need to call their constructor when adding them to the sycl::property_list`.

property_list q_props{property::queue::in_order(),
                      property::queue::enable_profiling()};
sycl::queue   q{q_props};

Next we use the parallel_for method of sycl::queue to submit our SYCL kernel for execution on the SYCL device. As the name implies, parallel_for method acts as a typical for loop except that all the loop iterates are done in parallel. The iteration domain or the range is represented by a sycl::range<N> object where the parameter N should equal 1, 2, or 3. For our simple hello world kernel, we only need a single dimensional iteration domain.

auto iter_domain = range<1>{N};

parallel_for method also accepts the code to be executed at each for iterate as a C++ lambda function. This function must be parameterized by a sycl::item<N> object to identify the execution instance at each point of the sycl::range<N>. We print the index of the for execution instance inside the kernel by calling the printf function provided as an experimental extension of the oneAPI compiler.

auto kernel = [=](sycl::item<1> idx) {
  ext::oneapi::experimental::printf("Hello from rank = %d\n", idx[0]);
};

Finally, we submit (dispatch) the kernel to the queue for execution. The parallel_for method returns a sycl::event object after that submission of the kernel. In this particular case, sycl::event object represents the status of the kernel that is being executed by the SYCL runtime. Since the SYCL queues execute kernels asynchronously, we can use the wait()` method of sycl::event object to wait till kernel completed.

auto event = q.parallel_for(iter_domain, kernel);
event.wait();

Note that we didn’t specify a SYCL device for the queue we created. In this case, SYCL runtime chooses a default device. We can get the sycl::device associated with the queue using get_device() method the queue. At the end of our example, we print the name of the device associated with the queue. We will see later how to select a device when constructing a queue.

std::cout << "The kernel executed on: "
          << q.get_device().get_info<info::device::name>();
